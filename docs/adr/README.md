# Architecture Decision Records

Each significant architectural decision is recorded here. The format is short and consistent: title, status, context, decision, consequences. Don't update past ADRs to reflect changes; write a new ADR superseding the old one.

The point of an ADR is to explain *why* a decision was made, so that future contributors (including future-you) understand the reasoning when evaluating whether to change it. An ADR without reasoning is just history.

## Format

```
# ADR-NNN: Title

## Status
[Proposed | Accepted | Superseded by ADR-MMM]

## Context
What was the problem? What constraints applied?

## Decision
What did we decide?

## Consequences
What follows from this? What does it cost us? What does it enable?
```

## Numbering

Sequential, zero-padded to 3 digits. Don't skip numbers. If an ADR is rejected, keep the file with status "Rejected" rather than deleting.
